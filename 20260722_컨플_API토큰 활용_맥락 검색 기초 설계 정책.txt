전체 Confluence 문서를 통합 검색하여 최신성 기반 답변 도출, 운영정책 정상 여부 판별, 의사결정 히스토리(배경) 추적, 잠재적 리스크(문제 소지) 검출이라는 4가지 목적을 동시에 달성하려면 단순한 질의응답 시스템을 넘어 '에이전틱 RAG(Agentic RAG)와 히스토리 추적 파이프라인'이 결합된 아키텍처를 구축해야 합니다.
Atlassian Rovo가 이 작업에서 만족스럽지 못했던 이유는 Confluence 문서의 작성 날짜(최신성) 가중치 제어가 불가능하고, 여러 문서 간의 인과관계(A 결정이 B 정책에 미친 영향)를 논리적으로 추론하는 기능이 부족하기 때문입니다. 이를 코드로 완벽히 제어하여 목적을 달성하는 구체적인 아키텍처와 구현 프로세스를 안내합니다.
🛠️ 핵심 시스템 아키텍처 요약
이 시스템은 총 4개의 레이어로 구성된 파이썬 기반 커스텀 RAG 파이프라인으로 구현됩니다.

[1. Confluence API 수집] ➔ [2. 메타데이터 임베딩 DB] ➔ [3. AI Agent (Multi-Step)] ➔ [4. 최종 리스크 보고서 결과]
(본문 + 수정일 + 작성자)    (날짜 가중치 부여 검색)      (정책 판별 & 히스토리 추적)     (정상 여부, 히스토리, 리스크)
1단계: 최신성(Recency) 중심의 데이터 구조 설계 (Data Ingestion)
Confluence API로 문서를 수집할 때 본문뿐만 아니라 '최종 수정일(lastUpdated)'과 '변경 이력(History)'을 반드시 메타데이터로 함께 추출하여 벡터 DB에 저장해야 합니다.


동작 방식: API 호출 시 version 및 history 필드를 포함하여 가장 최근에 업데이트된 문서가 무엇인지 명시적으로 태깅합니다.
파이썬 데이터 정제 예시:
# Confluence 페이지 상세 및 버전 정보 추출 page_id = "123456" page_detail = confluence.get_page_by_id(page_id, expand='body.storage,version,history')

metadata = {
    "page_id": page_id,
    "title": page_detail['title'],
    "last_updated": page_detail['version']['when'],  # 최신성 정렬의 기준 점수
    "author": page_detail['version']['by']['displayName'],
    "space": page_detail['_links']['webui'].split('/')[2]
}
text_content = page_detail['body']['storage']['value']
2단계: 최신성 기반 하이브리드 검색 (Recency-Weighted Search)
단순한 문맥 유사도 검색만 사용하면 3년 전 작성된 낡은 정책 가이드가 상위에 노출될 수 있습니다. 이를 막기 위해 Recency Reranking 또는 Metadata Filtering 기법을 적용합니다.


해결 방식:
 
시간 필터링: 벡터 DB(예: LangChain + Chroma/Pinecone) 조회 시, 최근 6개월~1년 이내에 업데이트된 문서에 검색 가중치 점수를 추가(Recency Boosting)합니다.
버전 추적: 동일한 제목이나 키워드의 정책 문서가 여러 개 존재할 경우, last_updated가 가장 최신인 문서만 LLM 프롬프트의 '최신 정책 정보' 컨텍스트로 주입합니다.
3단계: 정책 판별 및 의사결정 추적을 위한 에이전트 설계 (Agentic Thinking)
사용자의 질문을 받고 단순히 답변하는 것이 아니라, 내부적으로 단계별 추론(Chain-of-Thought)을 거치도록 Prompt와 LangGraph(또는 태스크 파이프라인)를 코드로 제어해야 합니다. AI에게 부여할 핵심 역할과 프롬프트 구조는 다음과 같습니다. [1] 
📝 시스템 프롬프트(System Prompt) 구조 예시

당신은 회사의 리스크 관리 및 운영정책 감사 AI 에이전트입니다. 
전달받은 Confluence 검색 결과를 바탕으로 반드시 다음 4단계 구조에 맞춰 답변을 작성하세요.

1. [최신 정책 기반 답변]
   - 질문에 대해 검색된 가장 최신(last_updated 기준) 문서를 바탕으로 명확한 사실 위주로 답변하세요.
   - 참조한 Confluence 문서 제목과 업데이트 날짜를 명시하세요.

2. [운영정책 정상/비정상 판별]
   - 현재 작동 중인 프로세스나 규정이 최신 운영정책 가이드라인과 완벽히 일치하는지 비교하여 [정상] 또는 [비정상]으로 판별하세요.
   - 불일치하거나 상충되는 규정이 있다면 그 지점을 명확히 짚어내세요.

3. [의사결정 히스토리 추적]
   - 이 정책이나 결과가 도출된 배경(과거 회의록, 기획서, 경영진 의사결정 문서 등)을 검색 결과에서 찾아내어 인과관계를 설명하세요.
   - 예: "2025년 10월 서비스 기획 회의록(ID: 999)에 따라 보안 규정이 강화되면서 이 정책이 도입됨."

4. [잠재적 리스크 및 문제 소지 검출]
   - 정책의 사각지대, 휴먼 에러 가능성, 상충되는 타 부서 정책, 혹은 1년 이상 업데이트되지 않아 현실과 동떨어진 규정 등 '문제 소지가 있는 지점'을 무조건 1개 이상 도출하여 경고하세요.
4단계: '문제 소지 검출'을 위한 다중 문서 교차 검증 (Cross-Reference)
하나의 정책만 보면 정상 같지만, 다른 부서의 문서나 과거 회의록과 대조하면 충돌이 발생할 수 있습니다. AI 에이전트가 이를 잡아내도록 하려면 다중 쿼리 검색(Multi-Query Retrieval) 로직을 구현해야 합니다.


구현 방법:
 
사용자가 "현재 환불 정책 프로세스가 정상인가요?"라고 질문합니다.
AI 내부적으로 질문을 쪼개어 다중 검색을 수행합니다.
 
쿼리 A: 최신 환불 운영 정책 가이드라인
쿼리 B: 최신 CS 환불 처리 프로세스 매뉴얼
쿼리 C: 재무팀 환불 정산 기준
세 가지 검색 결과를 하나의 프롬프트에 몰아넣고 "가이드라인(A)과 실제 매뉴얼(B), 정산 기준(C) 사이에 상충되는 모순점(리스크)이 있는지 검출하라"고 LLM에 명령합니다.
이를 통해 정책 간의 불일치로 인한 '비정상 상태'나 '문제 소지 지점'을 정확하게 짚어낼 수 있습니다.
